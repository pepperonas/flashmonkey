# LKS32MC081C8T8 — Facts extracted from datasheet

Source: `LKS32MC08x_Datasheet_EN_v1.93.pdf` (Linko Semiconductor, Jan 2023), in this directory.

## Device

| | |
|---|---|
| Core | ARM Cortex-M0, 96 MHz |
| Flash | 64 KB (C8 suffix) + 1 KB NVR |
| SRAM | 8 KB |
| Package | **TQFP48** (not LQFP48) |
| Full name | LKS32MC081C8T8(B) |

## Programming interface — only SWD

The datasheet describes **no UART ISP bootloader**, no "system memory" or mask-ROM programming region. Flash is programmed via **SWD only**, either with standard ARM debug tools (Keil-style) or Linko's proprietary "offline downloader". This matches RoboCoffee's empirical finding on the same chip in Xiaomi 3 Lite.

### SWD pin locations (TQFP48, LKS32MC081C8T8)

| Pin | Name | Function | Pull-up |
|---|---|---|---|
| **2** | RSTN / P0.2 | Reset (external) | 100 k internal |
| **47** | SWCLK | SWD clock | 10 k internal |
| **48** | SWDIO | SWD data | 10 k internal |

These pull-ups are **enabled automatically after power-up** — no external pull-ups needed to attach an ST-Link.

### Recommended external circuit on RSTN
> "Add a 100nF ground capacitor, plus a 100k built-in pull-up resistor. A 10k~20k pull-up resistor between AVDD and RSTN on PCB is recommended. Use a 100nF capacitor if the pull-up resistor is present."

## Flash protection mechanisms

Two *software-level* protection schemes. Neither is an irreversible fuse like STM32 RDP.

### 1. "Flash data anti-theft" (Section 16.1)
> "Flash data anti-theft by programming the last word of flash to any words other than `0xFFFFFFFF`"

The last 4 bytes of the 64 KB flash (i.e. bytes `0xFFFC–0xFFFF` within the main flash region) act as a key. If they are still erased (`FF FF FF FF`), read-back is plain. Any non-erased value enables anti-theft — readback presumably returns obfuscated or zeroed data.

**Consequence:** with SWD + mass-erase we can always erase + rewrite. With SWD + read-only attempt, if the manufacturer set this word, the dump may be scrambled.

### 2. SWD-pin GPIO multiplexing (Section 22)
SWDIO (= P2.15) and SWCLK can be reconfigured as general-purpose GPIOs after boot. If firmware enables this:
> "Debug and erase download functions cannot access the chip since SWD are now general GPIO"

**Bypass (described in the datasheet itself):** connect-under-reset within ~100 ms. The datasheet recommends:
> "leave a certain margin before switching SWD to GPIO, such as about 100ms, to ensure that the offline downloader can erase the chip and prevent the deadlock"

`openocd` and `pyocd` support connect-under-reset; this is the standard mitigation.

## Flash erase/program details (Section 16.1)

| Parameter | Value |
|---|---|
| Sector size | 512 B |
| Sector erase time (max) | 5 ms |
| Byte program time (max) | 7.5 µs |
| Endurance (min) | 20 000 cycles |
| Retention | > 100 years |

Sector erase is mandatory before re-programming any bytes in that sector.

## What this means for the Navee ST3 Pro BLDC flashing effort

| Question | Answer from datasheet |
|---|---|
| Can we flash the MCU via UART without SWD? | **No** — no UART ISP bootloader exists |
| Can we flash it via SWD with ST-Link v2? | **Yes**, if pins 47/48 + RSTN (pin 2) are accessible on the PCB |
| Is readout protection hardware-enforced? | **No** — only soft "last-word anti-theft" + optional GPIO muxing |
| Can a locked chip be mass-erased? | **Yes** via connect-under-reset + mass erase |
| What's the bypass for GPIO-muxed SWD? | connect-under-reset within ~100 ms window (standard pyocd/openocd feature) |
| PCB pins to find | Three tiny pads: RSTN (P0.2, MCU pin 2), SWCLK (pin 47), SWDIO (pin 48) |

Matches RoboCoffee's finding on the Xiaomi 3 Lite (same chip, same OEM Brightway): dump was possible without protection bypass, because the factory left both anti-theft mechanisms unused.

## Next physical steps

1. Locate three traces coming from MCU pins 2, 47, 48 on the ESC board — they terminate at either via-pads or test pads near the MCU.
2. Connect ST-Link v2: GND common, SWCLK→pin 47-trace, SWDIO→pin 48-trace, optional RSTN→pin 2-trace.
3. Probe:
   ```bash
   pyocd list --probes
   pyocd cmd -t cortex_m
   > reg
   ```
4. If no IDCODE response with normal mode: retry with `--connect=under-reset` and cycle power while holding RSTN low via the probe.
5. On success: `pyocd cmd -t cortex_m --flash-read main.bin 0 65536` for full flash dump.
